1. System Architecture — What It Is

The project is a two-part AI system: a Python backend that runs all AI algorithms, and a browser-based 3D cricket game as the frontend. The human player interacts with the game in the browser, while the AI player's decisions are made entirely by Python.

1.1 Why This Architecture?

  • Python backend — All 4 syllabus AI units (Search, CSP/Minimax, Bayesian/MDP, Q-Learning) require real algorithmic computation. Python has excellent libraries and makes it easy to implement these cleanly.
  • Browser frontend (HTML + Three.js) — The 3D cricket game is visualised in the browser so it works on any machine without installing a game engine. The human uses keyboard and mouse to play.
  • Flask HTTP server — The browser sends HTTP requests to Python for every AI decision. Flask is a minimal Python web framework that handles these cleanly.

1.2 How They Connect

👤
Human Player (Browser)

Presses SPACE to bat or clicks pitch to bowl — the game captures this input.

📡
HTTP Request (fetch API)

Browser sends JSON data to http://127.0.0.1:8888/bowl_decision or /shot_decision asking what the AI should do.

🐍

Receives the request, runs multiple AI algorithms, and returns a JSON response with the optimal decision.

🎮

Browser receives the AI decision and animates the ball, batsman, fielders accordingly. Score updates and commentary show what the AI decided.

2. Step-by-Step Game Flow

Every time the ball is bowled, the game follows this exact sequence of steps. Each step calls one or more AI algorithms from the syllabus.

1
Coin Toss
Human calls Heads or Tails. Python generates a uniform random outcome. Winner chooses to Bat First or Bowl First. This determines the role assignment for each innings.
2
Innings Begins — Role Assignment
If the human bats first: AI bowls (Python decides every delivery). If the human bowls first: AI bats (Python decides every shot). After the first innings, roles swap and a target is set.
3
AI Decides What to Bowl [When Human is Batting]
Before each ball, the browser sends the current match state (balls remaining, runs to defend, wickets taken, human's shot history) to Python. Python runs 4 algorithms in sequence:
OrderAlgorithmWhat It Decides
3aMDP (Unit III)High-level strategy: aggressive / defensive / balanced
3bMinimax α-β (Unit II)Exact delivery type: fast yorker / off-spin / bouncer etc.
3cBayesian Network (Unit III)Predicts what shot the human is likely to play
3dCSP Backtracking (Unit II)Places 9 fielders in optimal zones based on prediction
4
Human Bats — Timing & Direction
Ball animates from bowler to pitch. Human presses SPACE when the cursor is in the sweet spot of the timing bar. Direction is set with arrow keys or joystick. Shot type (drive/pull/cut/sweep/flick/loft) is selected with keys 1–6.
5
Cricket Rules Engine — Outcome Calculation
Based on timing quality, power, loft, and ball line, the engine applies real cricket laws: Law 22 (Wide), Law 21 (No Ball + Free Hit), Law 19 (Boundary), Law 30 (Bowled), Law 33 (Caught), Law 36 (LBW), Law 38 (Run Out). Result: 0–6 runs or a wicket.
6
Q-Learning Update [Unit IV]
After each ball, the browser sends the outcome to Python's /learn_update endpoint. Python updates the Q-table using the Bellman equation so the AI improves its batting policy over time.
7
AI Bats [When Human is Bowling]
Human clicks the pitch to aim and deliver. Python receives the delivery and runs Q-Learning + Minimax + DFS to decide the best shot for the AI batsman (Travis Head, Abhishek Sharma, etc.). The AI plays the shot with animated movement and commentary.
8
End of Innings — Target Set
After the set number of overs (1/3/5) or 10 wickets, the innings ends. The target for the second innings = first innings score + 1. Wagon wheel is displayed for human batting innings.
9
Final Scorecard
After both innings, the winner is declared. A detailed scorecard shows runs, wickets, run rate, and lists all 4 AI algorithms used during the match. The Q-table is saved for the next session so the AI has learned from this game.

3. Unit I — Search Algorithms

Unit I BFS · DFS · A* · Heuristics · Problem Formulation

3.1 A* Search — Fielder Path Planning

When the ball is hit, A* finds the shortest path for a fielder to run from their current position to the ball's landing zone. The field is modelled as a 2D grid of 2-metre cells within the 35-metre boundary circle.

BFS would explore all cells at equal cost — too slow for a large field. DFS would find a path but not the shortest one. A* uses a heuristic (Euclidean distance to the ball) to guide the search towards the goal first, making it significantly faster while still guaranteeing the optimal path.

A* Algorithm (from search.py)

f(n) = g(n) + h(n)
  • g(n) = actual distance the fielder has travelled so far (step cost)
  • h(n) = Euclidean distance from current cell to ball landing zone (heuristic)
  • f(n) = estimated total path cost — the cell with lowest f is explored first
The AI Algorithm Log shows: "A* (fielder→ball): 7 nodes, h=3.2" — meaning 7 grid cells were explored to find the optimal route.

3.2 BFS — Fielder Coverage Zone

BFS explores all grid cells reachable by a fielder within a set number of steps, building a "coverage zone" — the area that fielder can reach to stop the ball.

We need to find ALL reachable cells within a maximum distance, not just the shortest path to one cell. BFS explores level by level (all cells at distance 1, then 2, etc.), which naturally gives us this "reachable zone" answer.

In the game: After the ball is hit, BFS determines which fielder is closest and can reach the ball — shown in the log as "BFS nearest fielder: 6 hops".

3.3 DFS Shot Decision Tree — AI Batting

When the AI (SRH batsman) needs to choose a shot, DFS traverses a decision tree based on the delivery characteristics: length (yorker/bouncer/good-length/full/short) and line (off/middle/leg stump).

The shot decision tree is naturally a tree structure. DFS explores one branch completely before backtracking — this matches how a batsman instinctively commits to a shot by following one decision path. The tree is small so DFS is efficient here.

4. Unit II — CSP & Game Search

Unit II CSP · Backtracking · Minimax · Alpha-Beta Pruning

4.1 Minimax with Alpha-Beta Pruning — Bowling Decision

When AI needs to bowl, Minimax searches a game tree where the AI (MAX player) trys to maximise wicket probability and economy, while assuming the human batsman (MIN player) will respond optimally. Alpha-Beta pruning removes branches that cannot affect the final decision, reducing computation by ~60–70%.

Cricket is a two-player adversarial game. The AI bowler and human batsman have opposing goals. Minimax is specifically designed for such zero-sum games — it guarantees the AI picks the delivery that is best even if the human plays perfectly.

Evaluation Function (from minimax.py)

value = wicket_probability × 15.0 + (5.0 − expected_runs) × 1.5
# Context modifiers:
if balls_left ≤ 2 and delivery == "yorker":
    wicket_value × 1.6    # Yorker is deadliest at death overs
if pitch_spin:
    wicket_value × 1.3    # Spin on a turning pitch
"Minimax α-β (depth=5) chose FAST delivery — 80 game-states evaluated, v=3.42 > alt=1.8"

4.2 Minimax — Batting Decision (AI Batsman)

When AI (Travis Head, Klaasen, etc.) is batting, Minimax also selects the best shot. The evaluation considers: expected runs, wicket risk, and field placement. If fielders block a particular direction, the score for that shot is reduced.

value = expected_runs − wicket_risk × 10.0 − (fielder_coverage_penalty × 1.2)

4.3 CSP — Field Placement

After the Bayesian Network predicts which shot the human will play, CSP places 9 fielders into optimal zones. The CSP has 9 variables (F1–F9), 23 possible zone values, and 2 constraints:

  1. No two fielders in the same zone (uniqueness constraint)
  2. At least one slip fielder (structural constraint)

Field placement is a classic constraint satisfaction problem — we have limited slots, limited zones, and rules about what is and isn't allowed. CSP backtracking with MRV (Minimum Remaining Values) and LCV (Least Constraining Value) ordering solves this optimally in just milliseconds.

5. Unit III — Bayesian Networks & MDP

Unit III Bayesian Networks · CPTs · Uncertainty · MDP · Value Iteration

5.1 Bayesian Network — Shot Prediction

Before each ball, the Bayesian Network predicts the probability distribution over what shot the human batsman will play. This is used by CSP to set the field and by Minimax to choose the right delivery. The network also updates its belief after every ball based on what shot the human actually played.

There is uncertainty about what shot the human will play. A Bayesian Network models this uncertainty using conditional probability tables (CPTs) — it gives not just one prediction but a probability for each shot. As the human reveals their preferences over the innings, the network updates its beliefs (Bayesian belief update).

Network Structure

NodeRoleExample
DeliveryTypeParent node — type of ballgood_length_off, yorker, bouncer
ShotTypeChild node — predicted shotdrive (38%), defend (26%), cut (22%)
ShotDirectionChild of ShotType — ball directioncover (40%), mid-off (20%)

Bayesian Update Rule (from bayesian.py)

P(Shot | Delivery, History) ∝ P(History | Shot) · P(Shot | Delivery)

After each observed ball, the likelihood P(History | Shot) is updated using Laplace smoothing to avoid zero probabilities:

P_posterior(shot) = P_prior(shot) × (count + 1) / (total_obs + |shots|)
"Bayesian: P(drive|good_length_off)=0.38, P(boundary|fast,spd=140)=0.18 — field adjusted"

5.2 Markov Decision Process (MDP) — Match Strategy

Before each over, the MDP determines the AI's bowling strategy: should it bowl aggressively (targeting wickets) or defensively (minimising runs)? This is the high-level decision that then guides Minimax's delivery selection.

The match situation evolves over time — balls remaining, runs to defend, wickets in hand all change every ball. MDP explicitly models this sequential decision-making under uncertainty. Value Iteration computes the optimal policy π*(s) for every possible game state before the match even starts.

MDP Components

ComponentDescriptionIn Cricket
States (S)All possible game situations(balls_remaining, runs_to_defend, wickets_taken)
Actions (A)Available strategiesaggressive / defensive / balanced / tricky
Transitions (T)P(s' | s, a) — stochastic outcomesP(wicket | fast_aggressive) = 0.28
Rewards (R)R(s, a, s')+wicket gets high reward, +boundary is penalty
Policy π*Optimal action for each statee.g., "last 2 balls, 10 to defend → AGGRESSIVE"

Value Iteration (runs at startup)

V*(s) = max_a [R(s,a) + γ · Σ T(s,a,s') · V*(s')]

The server runs Value Iteration when it starts up (converges in 7 iterations) and stores the optimal value function V*(s) for all states. During the match, looking up the policy for the current state is instant.

"MDP: V*(3 balls left, chase 12)=4.2 → π*=aggressive — Minimax will use fast yorker"

6. Unit IV — Q-Learning (Reinforcement Learning)

Unit IV Q-Learning · ε-greedy · Bellman Equation · Policy Learning

6.1 What Q-Learning Does in This Game

Q-Learning is the AI batting agent's learning system. The AI batsman (Travis Head, Klaasen, etc.) learns which shot to play against each type of delivery over time. It starts by exploring randomly and, as it accumulates experience, increasingly exploits the best shots it has learned.

Unlike the other algorithms, Q-Learning requires no pre-programmed knowledge about which shot is best. It learns purely through trial and error — playing shots, observing the runs scored (reward), and updating its policy. This is the essence of Reinforcement Learning: the agent learns from interacting with the environment.

6.3 Bellman Update — How the AI Learns

After every ball, the Q-table is updated using the Bellman equation:

Q(s, a) ← Q(s, a) + α · [r + γ · max_a' Q(s', a') − Q(s, a)]
  • α = 0.12 — Learning rate: how much to update Q from new information
  • γ = 0.90 — Discount factor: future rewards matter but less than immediate ones
  • r — Reward from this ball (runs scored or wicket penalty)
  • max_a' Q(s', a') — Best future Q-value (Bellman backup)

6.4 ε-Greedy Exploration

  • With probability ε — explore: pick a random valid shot (try something new)
  • With probability 1 − ε — exploit: pick argmax Q(s, a) (use best learned shot)

If the AI always exploits, it might get stuck in a suboptimal policy (e.g., always driving when pulling is better against short balls). ε forces it to occasionally try other shots and discover better options. ε decays from 0.35 to 0.05 over time — early innings = more exploration, later = more exploitation.

6.5 Q-Table Persistence

The Q-table is saved to q_table_batting.json after every 6 balls (one over). This means the AI remembers what it learned in previous games. If you play multiple matches, the AI gets progressively smarter — fewer random shots, better decision-making.
